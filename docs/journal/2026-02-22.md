# Dev Journal — February 22, 2026

## Day 2: Gateway

---

**What I worked on:**

Built and validated Gateway — the AI protection layer. Merged two separate projects (Prompt-Shield and Prompt-Firewall) into a single clean codebase, tuned the detection engine, and confirmed the full pipeline works end to end.

---

**What was built:**

Gateway is a FastAPI service running on port 8001. Any AI deployment routes prompts through it before sending them to the model. The service inspects every prompt and returns a decision: allow, block, sanitize, or log.

File structure mirrors ARIA exactly:

```
gateway/
├── core/
│   ├── detection_engine.py   ← Three-layer ensemble detector
│   ├── policy_engine.py      ← Configurable rule evaluation
│   ├── sanitizer.py          ← Malicious pattern removal + PII redaction
│   ├── models.py             ← Shared data structures
│   └── websocket_manager.py  ← Real-time dashboard event streaming
├── routes/
│   ├── inspect.py            ← POST /inspect — primary endpoint
│   ├── admin.py              ← Policy management API
│   └── health.py             ← GET /health
├── policies/
│   └── default.yaml          ← Configurable ruleset
└── main.py
```

---

**The merge decision:**

Prompt-Firewall had the better architecture — policy engine, sanitizer, models, and a proper orchestration layer. That became the skeleton.

Prompt-Shield had the better detection logic — a three-layer ensemble (rule-based, statistical, semantic) with weighted scoring. That became `detection_engine.py`.

Prompt-Firewall's old `detector.py` was deleted entirely. It was just a wrapper that tried to import Prompt-Shield as an external package and fell back to basic pattern matching. No reason to keep it now that the ensemble lives directly in the codebase.

Two things were added that didn't exist in either original project:
- `websocket_manager.py` — mirrors ARIA's exactly so the unified dashboard gets the same event format from both components
- `routes/admin.py` — full CRUD policy management at runtime via API, no YAML edits or restarts required

---

**Detection engine architecture:**

Three layers combined with weighted scoring:

```
Rule-Based   (40%) — pattern matching for known attacks
Statistical  (25%) — anomaly detection on prompt characteristics  
Semantic     (35%) — intent analysis, imperative detection, authority claims

Final score = (0.40 × rule) + (0.25 × statistical) + (0.35 × semantic)
Hard override: if rule_score >= 45, ensemble = max(ensemble, rule_score × 0.80)
```

Detection categories: instruction override, role manipulation, prompt leaking, jailbreak, context manipulation, code injection, data exfiltration, model extraction.

---

**Key decisions made and why:**

**Hard override on strong rule matches:**
The original scoring let a zero statistical score bury an obvious injection. "Ignore all previous instructions" was scoring 17.1 — below threshold, action: allow. That's a failure. The fix: if the rule layer alone hits 45+, the ensemble is forced to at least `rule_score × 0.80`. A clean-looking prompt can't escape detection if it matches a high-confidence pattern.

**First hit scores 70, not 30:**
Original scoring gave each pattern hit 30 points. A single "ignore all previous instructions" match scored 30 out of 100 on the rule layer. Changed to: first hit = 70, each additional = +15, cap at 100. One clear injection pattern is a strong signal, not a weak one.

**Threshold lowered from 50 → 40:**
Slightly more sensitive overall. Reduces the chance of obvious attacks slipping through at the boundary.

**No Pydantic in routes:**
FastAPI's internal OpenAPI models conflict with Pydantic v1, and Pydantic v2 requires Rust to compile from source. Norton flagged the Rust build toolchain as malware (false positive — EvoGen classification). Rather than fight the AV, routes use `Request` + `await request.json()` directly. No Pydantic dependency in the request handling layer at all. Core data structures use Python dataclasses instead.

---

**Problems encountered:**

**Pydantic v2 / Rust / Norton:**
Pydantic v2 needs Rust to compile pydantic-core from source. Rust was installed via winget. Norton quarantined `build-script-build.exe` (the Rust build runner) as EvoGen malware — false positive, but unrecoverable mid-install. Attempted Pydantic v1 as workaround — FastAPI's internal OpenAPI model has a field literally named `name` which Pydantic v1 can't infer a type for, crashing on startup. Resolution: removed Pydantic from routes entirely, pinned to a version of pydantic+fastapi with available pre-built Windows wheels, disabled Swagger docs (`docs_url=None`) to prevent the OpenAPI model from loading.

**Import error on startup:**
`cannot import name 'get_detection_engine' from 'core'` — `routes/__init__.py` was missing. FastAPI couldn't resolve the `core` package without it. Fixed by creating the empty `__init__.py`.

**Wrong uvicorn executable:**
Running `uvicorn` directly was hitting the system Python installation, not the venv. Fixed with `python -m uvicorn` which forces the active venv's interpreter.

**Validation:**

Before tuning:
```

```

After tuning:
```
threat_score: 61.6

```

Detection identified the injection, sanitizer stripped it, harmless remainder passed through. 5ms response time.

**Debugging notes:**

- `python -m uvicorn` over bare `uvicorn` — always use the module invocation in venv environments on Windows
- Norton will flag Rust build artifacts — add `C:\Users\<user>\.cargo\` and `C:\Users\<user>\.rustup\` to exclusions before installing Rust
- Pydantic v1 + FastAPI: the `name` field conflict is in FastAPI's internal `Contact` model, not user code — removing Pydantic from routes entirely is cleaner than trying to version-pin around it
- `routes/__init__.py` must exist even if empty — Python won't resolve the package without it

## Day 2 (Part 2): Shared Threat Intelligence Layer

---

**What I worked on:**

Wired ARIA and Gateway into a shared threat intelligence layer. Both components now write to a single SQLite database and publish to a shared event bus. Cross-vector correlation is live — the same attacker operating across voice and prompt vectors gets a unified profile.

---

**What was built:**

Three files in `shared/`:

```
shared/
├── models.py        ← Unified ThreatEvent schema
├── threat_intel.py  ← SQLite DB — single source of truth
├── event_bus.py     ← Async pub/sub for real-time cross-component awareness
└── __init__.py
```

Two integration patches:
- `gateway/routes/inspect.py` — writes ThreatEvent to shared DB after every malicious detection
- `aria/routes/voice.py` — writes ThreatEvent to shared DB after every confirmed vishing attempt

---

**Shared schema design:**

The core challenge was designing a schema that accommodates two fundamentally different threat types — voice social engineering (ARIA) and prompt injection (Gateway) — without forcing either component to change its internal data structures.

The solution was a `ThreatEvent` dataclass with shared fields that both can populate:

```
Correlation:   session_id, caller_id, user_id, ip_address
Assessment:    threat_score, threat_level, is_malicious, confidence
Attack data:   categories, techniques, mitre_tags, action_taken
Content:       raw_content, sanitized_content, primary_objective, evidence
Extras:        component_metadata (component-specific details that don't fit)
```

ARIA populates `caller_id`, `techniques`, `mitre_tags`, `evidence`, `primary_objective`.
Gateway populates `categories`, `sanitized_content`, `method_scores` (in component_metadata).
Both populate `session_id`, `threat_score`, `threat_level`, `action_taken`.

---

**Cross-vector correlation:**

Every `write_event()` call automatically runs `_correlate()` in the same DB connection. It checks if the same `session_id` has appeared from a different source. If it has, it creates or updates an `AttackerProfile` — a unified record linking all events from that actor across ARIA and Gateway.

```
threat_events table     → every individual detection
attacker_profiles table → cross-vector actor records
```

The `get_cross_vector_sessions()` query finds any session that appears in both `aria` and `gateway` events — highest priority alerts.

---

**Event bus:**

Each component publishes to a shared `EventBus` after writing to the DB. Subscribers get an asyncio Queue and receive all events in real time. The unified dashboard will subscribe to this bus and receive both ARIA and Gateway events through a single WebSocket connection.

```python
# Publishing (both components do this after detection):
await get_event_bus().publish_threat("gateway", event.to_dict())
await get_event_bus().publish_threat("aria", event.to_dict())

# Subscribing (unified dashboard will do this):
queue = await bus.subscribe()
event_type, data = await queue.get()
```

---

**Problems encountered:**

**`cannot access local variable 'session'`:**
The `_write_aria_event_to_intel()` call was placed before `session = active_sessions[session_id]`, so `session.get("caller")` crashed. Fixed by moving the call to after the session assignment.

**`No module named 'shared'`:**
Running the DB verification command from inside `gateway/` instead of the monorepo root. `shared/` lives at `corvus-mirage/shared/` — must be run from that root.

**Alert webhook failure:**
`Failed to send alert to webhook: Request URL is missing an 'http://' or 'https://' protocol.` — blank webhook URL in `.env`. Not related to shared intel, does not affect detection or DB writes. Non-critical, fix later.

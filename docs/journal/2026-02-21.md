## Day 1: The Beginning

---

**What I worked on:**

No code today. Today was about vision, naming, and architecture. Sometimes the most important work happens before a single file is created. Established the full concept for Corvus Mirage — an AI security platform designed to protect both AI systems and the humans who use them.

---

**How we got here:**

Started by evaluating Shannon (KeygraphHQ) as a benchmark — a fully autonomous AI pentester achieving 96.15% on the XBOW benchmark. The question was whether the Red Team Attack Simulator could compete. The honest answer: not yet, and not by trying to replicate Shannon directly.

The realization: Shannon protects apps *from* attackers. Nobody has built something that protects *AI systems themselves* AND defends against the human attack vector (social engineering) in real time. That's the gap. That's Corvus Mirage.

---

**Key decisions made and why:**

**Name: Corvus Mirage**
- Corvus = Latin for raven. Intelligent, watchful, sees everything.
- Mirage = What the attacker sees. An illusion that pulls them in.
- Together: *We see you clearly. You see nothing real.*
- Rejected: Umbra Mirage (too dark/intimidating for enterprise buyers), Wraith, Vigil, Bastion.

**Architecture: Three-component platform under one roof**
- ARIA — Honeypot + Vishing Detection & Response (the face)
- Gateway — AI Protection Layer (the shield)
- Red Team Simulator — Validation Engine (the proof)

**Why vishing specifically:**
- MGM Resorts 2023 breach started with a 10-minute phone call. $100M in damages.
- Scattered Spider used vishing as initial access. No real-time AI detection layer exists for this.
- Companies are already deploying AI to answer calls. ARIA sits there, detects the attack, and notifies the human rep mid-call with a live transcript and threat profile.

**Why start from ARIA:**
- Dashboard is already enterprise-grade. MITRE ATT&CK tagging, attacker profiling, session correlation, threat scoring — all functional.
- Extending voice is adding an input layer to what already works, not rebuilding.
- Highest novelty. Most differentiating. Build while the idea is hot.

**Why build from scratch (mostly):**
- Prompt-Shield and Prompt-Firewall exist but are separate codebases solving adjacent problems. Fragmentation kills pitchability. Merge into Gateway.
- Monorepo approach — one install, three components, unified dashboard.
- Pull proven components from existing projects rather than rewrite everything.

**Documentation strategy:**
- Two parallel tracks: Technical Docs (external facing) and Dev Journal (timeline/decision log).
- Journal starts today even though no code was written. Product decisions are real work.

---

**The pitch angle:**

Corvus Mirage is not trying to be Shannon. Shannon is an offensive tool for finding vulnerabilities in web apps. Corvus Mirage is a defensive platform for securing AI deployments and the humans operating them. Shannon has no answer for vishing. Shannon has no answer for prompt injection at scale across multiple AI systems. That's the differentiator.

Primary target: Anthropic. The platform is built using Claude SDK and API. It protects AI systems broadly. It's a natural conversation.

Secondary targets: Enterprise security companies, SOC tool vendors, any org deploying AI at scale.

## End of Day Update

**Milestone: Full pipeline validated on Day 1**

The core of Corvus Mirage is working. On the first day of development, the entire voice detection pipeline was tested end to end with a live phone call.

**What was built today:**

ARIA's voice layer — from scratch:
- `main.py` — FastAPI entry point with WebSocket manager and auto-initialized threat intel DB
- `core/detection_engine.py` — Claude-powered analysis engine that takes a live transcript and returns structured threat assessment with techniques, MITRE tags, threat score, sophistication level, and recommended action
- `core/websocket_manager.py` — Real-time streaming layer pushing all events to the dashboard
- `routes/voice.py` — Twilio webhook handler and Deepgram streaming engine
- `routes/sessions.py` — Session query endpoints for dashboard
- `routes/health.py` — Health check endpoint

**What was tested:**

Live call to the Twilio number. Results:
- Twilio webhook hit `/voice/incoming` ✅
- WebSocket stream established ✅
- Deepgram transcribed voice in real time ✅
- Transcript fed into Claude detection engine ✅
- `POST https://api.anthropic.com/v1/messages 200 OK` ✅

The full loop — phone call → transcription → AI detection → response — worked on the first real test.

**Debugging notes:**
- Deepgram SDK `start()` is synchronous, not async — removed `await`
- `on_transcript` callback must be a regular `def`, not `async def` — Deepgram calls it from its own thread
- Used `asyncio.run_coroutine_threadsafe()` to bridge sync Deepgram callbacks into FastAPI's async event loop
- Used list-based nonlocal variables to avoid Python closure scoping issues

**Tools and services activated today:**
- ngrok — local tunnel for Twilio webhook during development
- Twilio trial account — phone number provisioned and webhook configured
- Deepgram — Member role API key, nova-2 model, real-time transcription confirmed working
